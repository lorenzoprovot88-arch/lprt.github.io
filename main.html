Situation : L'entreprise utilise un espace SharePoint couplé à un fichier Excel maître servant de base de données (BDD) pour inventorier l'ensemble du parc informatique (caractéristiques matérielles, affectations), que les postes appartiennent à l'entité principale ou à des filiales (SKYDOME, ECODIS, ESSEMES), en interne comme à l'externe.

Problématique : Selon l'emplacement réseau de la machine (bureau, télétravail hors VPN) ou son entité de rattachement, les protocoles et les droits d'accès pour interroger et remonter ces informations varient, rendant la centralisation complexe et hétérogène.

Mission : Standardiser le processus afin de mettre en œuvre une méthode unique, universelle et transparente pour collecter et unifier les données de tous les assets informatiques du groupe.

Architecture de la Solution (Le Plan Établi)

Étape 1 : Le Déclenchement (Déploiement en masse)

  • GPO (GPO_user_task_script_inventory.xml) : Déploiement d'une stratégie de groupe sur l'ensemble de l'Active Directory.
  • Tâche Planifiée (Inventory.xml) : La GPO inscrit une tâche planifiée locale sur chaque machine pour exécuter l'inventaire de façon récurrente.

Étape 2 : La Collecte Locale des Données

Le script EndUserAssets.vbs appelle silencieusement en arrière-plan un script PowerShell principal (EndUserAssets.ps1). Ce dernier requête le système (WMI/CIM) pour extraire toutes les caractéristiques de la machine (Nom d'hôte, CPU, RAM, OS, versions des agents de sécurité, numéro de série, entité, etc.).

Étape 3 : La Transmission des Données & Gestion du Réseau

Le script tente une écriture directe dans la base de données via un composant ADODB. Deux scénarios de routage s'exécutent automatiquement :

  • 🟢 Chemin A : Succès de la liaison ADODB
    Le PC est connecté au réseau interne ou sous VPN : les données sont injectées directement dans la liste SharePoint racine (End-User_Assets KLA-FR).
  • 🔴 Chemin B : Échec de la liaison ADODB (Machine externe / isolée)
    Si le PC est sur un réseau externe ou un sous-réseau bloqué :
    • L'exécutable d'assistance KlaInventoryHelper.exe prend automatiquement le relais.
    • Email Submission : Les données collectées sont encapsulées et transmises de manière sécurisée par e-mail (flux universellement ouvert).
    • Power Automate (TSK05 End-User Assets) : Un processus Cloud intercepte l'e-mail, extrait le contenu de l'inventaire et va l'inscrire automatiquement dans la liste SharePoint principale.

Étape 4 : Centralisation et Distribution Finale

Une fois les données stockées au sein de la liste SharePoint globale "KLA-FR" :

  • Elles se synchronisent avec le fichier Excel maître (EndUserAssets.xlsm).
  • Un mécanisme distribue ensuite les données triées vers les listes SharePoint dédiées de chaque filiale (Division - Sharepoint List).

Schéma du Flux (Process Flow)

GPO_user_task_script_inventory.xml (GPO) ↓ Inventory.xml (Scheduled Task) ↓ EndUserAssets.vbs ↓ EndUserAssets.ps1 ↓ ├─ 🟢 ADODB Success ➔ End-User_Assets (KLA-FR - Sharepoint List) └─ 🔴 ADODB Failure ↓ KlaInventoryHelper.exe ↓ Email Submission ↓ TSK05 End-User Assets - Record (Power Automate) ↓ End-User_Assets (KLA-FR - Sharepoint List) ↓ EndUserAssets.xlsm ↓ End-User Assets (Division - Sharepoint List)

Stockage local sur le Poste

Toutes les informations récoltées par le script sont sauvegardées localement dans un fichier texte brut dans le répertoire C:\KLA\EndUserAssets\. Ce script s'exécute de façon continue pour assurer la fraîcheur des données et faciliter le diagnostic en cas de panne.
